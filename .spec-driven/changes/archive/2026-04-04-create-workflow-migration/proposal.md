# Proposal: create-workflow-migration

## What
Migrate all backend workflow services from hand-written Java implementations to Lealone `CREATE WORKFLOW` DDL definitions with LLM-generated implementations. The ticket workflow is the first and current migration target. This establishes the platform-level convention that all workflow-enabled services MUST be defined via `CREATE WORKFLOW` DDL.

## Why
Lealone's new `CREATE WORKFLOW` feature allows workflow service implementations to be auto-generated by an LLM using existing service DDLs and table schemas as context. Adopting this pattern:
- Eliminates hand-written JDBC workflow boilerplate
- Makes workflow interfaces machine-readable and declarative (DDL as the single source of truth)
- Aligns the backend with the platform's AI-friendly fast-service philosophy
- Establishes a reusable pattern for future workflow-enabled entities

## Scope

### In scope
- Register `KanbanStateMachine` as a Lealone SERVICE in `services.sql` with methods for transition validation, so it is available as LLM context during workflow code generation
- Add `CREATE WORKFLOW ticket_workflow_service (getWorkflow, executeWorkflowAction)` to `services.sql` before `ticket_service`
- Remove `getWorkflow` and `executeWorkflowAction` from `ticket_service` DDL and from `TicketServiceImpl.java`
- Delete `TicketWorkflowService.java`
- If the LLM-generated JSON response shape does not align with the frontend `WorkflowInstance` contract, add a server-side adapter within the workflow service or a thin wrapper class â€” do not change the frontend
- Update frontend API call paths from `ticket_service/getWorkflow` and `ticket_service/executeWorkflowAction` to `ticket_workflow_service/getWorkflow` and `ticket_workflow_service/executeWorkflowAction`
- Write integration tests that verify the generated workflow service returns the correct JSON shape and enforces the correct state machine transitions
- Tests serve as the verification gate for generated code: when regeneration is needed (e.g. schema changes), delete the generated `.java`/`.class` files and re-run tests to confirm correctness
- Update `workflow-component.md` spec to add the convention: backend workflow services MUST be defined via `CREATE WORKFLOW` DDL

## Unchanged Behavior
- The frontend `workflow-panel.tsx` renders identically before and after this change
- The `WorkflowInstance` JSON contract received by the frontend remains identical
- Workflow actions (submit, approve, reject, reassign) and their state transition rules remain identical
- The `ticket_service` DDL continues to expose ticket CRUD methods (createTicket, moveTicket, listTicketsByProject)
- The `ticket_workflow_history` table schema is unchanged
