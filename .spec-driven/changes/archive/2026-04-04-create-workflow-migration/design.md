# Design: create-workflow-migration

## Approach

### Step 1: Register KanbanStateMachine as a Lealone SERVICE
Add a `CREATE SERVICE kanban_state_machine_service (...)` entry to `services.sql` with two methods:
- `isTransitionAllowed(fromState varchar, toState varchar) boolean`
- `ensureTransition(fromState varchar, toState varchar) void`

The implementing class is the existing `KanbanStateMachine`. A thin adapter (`KanbanStateMachineServiceImpl`) bridges the Lealone executor dispatch convention (lowercase method names) to the static utility methods. This service must appear in `services.sql` before `ticket_workflow_service` so its DDL is in the schema when `genWorkflowCode` runs.

### Step 2: Define ticket_workflow_service via CREATE WORKFLOW
Add to `services.sql`:
```sql
CREATE WORKFLOW ticket_workflow_service (
  getWorkflow(ticketId long) varchar,
  executeWorkflowAction(ticketId long, actionName varchar, actorUserId long, comment varchar, assigneeUserId long) varchar
)
PACKAGE @packageName
IMPLEMENT BY 'com.fastservice.platform.backend.ticket.TicketWorkflowServiceImpl'
GENERATE CODE @serviceSrcDir;
```

On first startup with `LLM_PROVIDER` configured, Lealone calls `genWorkflowCode`, which:
1. Collects all other SERVICE DDLs (including the new `kanban_state_machine_service`) as context
2. Lists all table schemas
3. Prompts the LLM to generate `TicketWorkflowServiceImpl` as a Java class orchestrating those services

### Step 3: Remove hand-written workflow code
- Delete `TicketWorkflowService.java`
- Remove `getWorkflow` and `executeWorkflowAction` from `ticket_service` DDL in `services.sql`
- Remove corresponding methods from `TicketServiceImpl.java`

### Step 4: JSON contract adapter (if needed)
After the LLM generates the implementation, run integration tests. If the returned JSON does not match `WorkflowInstance`, add a `TicketWorkflowJsonAdapter` class that the generated service delegates to for response formatting. The adapter is hand-written and version-controlled; the generated class calls it.

### Step 5: Frontend API path update
Update `frontend/src/lib/api/hooks.ts`:
- `ticket_service/getWorkflow` → `ticket_workflow_service/getWorkflow`
- `ticket_service/executeWorkflowAction` → `ticket_workflow_service/executeWorkflowAction`

### Step 6: Update workflow-component spec
Add a platform convention requirement to `workflow-component.md`: backend workflow services MUST be declared via `CREATE WORKFLOW` DDL and MUST produce JSON conforming to the `WorkflowInstance` contract.

## Key Decisions

### KanbanStateMachine as a service rather than inlining rules in DDL comments
Registering it as a Lealone SERVICE means `genWorkflowCode` automatically includes its full DDL as context. The LLM can then reference `KanbanStateMachineServiceImpl` by name and call its methods directly. Inline DDL comments are fragile and not machine-readable by the generator.

### Test-first generation gate
Integration tests are written to assert the JSON shape of `getWorkflow` and the state transition enforcement of `executeWorkflowAction` before any generated class is committed. If the class file is deleted and regenerated, tests re-verify the new generation. This replaces ad-hoc manual inspection.

### Adapter over regeneration loop
If the LLM generates almost-correct code but with a minor JSON key naming difference, a thin adapter is cheaper and more stable than iterating on the LLM prompt. The adapter is the seam between the generated implementation and the versioned contract.

### Workflow methods fully extracted from ticket_service
Moving `getWorkflow` and `executeWorkflowAction` into a dedicated `ticket_workflow_service` DDL makes the workflow boundary explicit and models the platform convention cleanly. The cost is a frontend API path change, which is acceptable within this change.

## Alternatives Considered

### Keep workflow methods in ticket_service, delegate internally
Would avoid the frontend API path change. Rejected because it obscures the architectural boundary — the `ticket_service` DDL would continue to expose methods not generated by its own code, which muddies the `CREATE SERVICE` vs `CREATE WORKFLOW` distinction.

### Use DDL comments to pass state machine rules to LLM
Simpler setup, no new service. Rejected because the generator ignores comment text — only DDL structure and table schemas are passed. The KanbanStateMachine logic would not reach the LLM context reliably.
