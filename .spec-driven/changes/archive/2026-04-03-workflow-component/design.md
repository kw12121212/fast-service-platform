# Design: workflow-component

## Approach

定义 `WorkflowDescriptor`、`WorkflowActionDescriptor` 和流程实例输入结构，由调用方传入当前状态、阶段列表、责任人信息、可执行动作、历史记录和必选评论能力。`WorkflowPanel` 根据这些声明式输入渲染统一的工作流界面，并在用户点击动作时把动作标识、评论内容和附加参数通过 `onAction` 回调交给调用方。组件不拥有状态机逻辑，不自行拉取数据，也不决定某个动作是否允许执行；这些责任继续保留在页面和现有 data-access 路径上。除组件本身外，当前 runnable admin frontend 还需要提供一个最小真实示例，用来证明工作流组件与真实 backend-backed 路径的集成方式。

## Key Decisions

- **工作流组件保持为渲染和交互收集层** - 这样可以复用现有 frontend data-access 边界，并避免组件演化成流程引擎。
- **V1 只覆盖单记录或单流程实例** - 企业内部最常见、最稳定的流程交互是围绕一条业务记录进行的，先把这个路径做稳，而不是扩到批处理或跨实体编排。
- **动作集合保持有界且包含 `reassign`** - 首版明确围绕提交、批准、驳回、重新分配等常见企业操作展开，减少 descriptor 膨胀风险，同时保证一个足够真实的企业流程动作集合。
- **评论与历史作为 V1 必备界面能力** - 这两者是企业流程可追踪性的最小可见组成部分，但它们仍然只消费调用方提供的数据，不产生新的后端契约假设。
- **当前 runnable frontend 必须提供一个真实示例** - 仅有组件 contract 不足以证明平台可复用性，首版需要最小真实集成路径，方便贡献者和 AI 理解如何接入。

## Alternatives Considered

- **把工作流留给各业务模块自行实现**：短期看更省事，但会把重复 UI 和交互边界分散到每个模块，不符合当前 milestone 的平台化目标。
- **直接设计成通用 BPM / orchestration 平台**：范围过大，会引入规则系统、分支建模、流程设计器和执行引擎等超出当前仓库边界的能力。
- **组件自己根据后端流程定义自动拉取和渲染**：会把数据访问、权限判断和执行规则耦合进组件内部，破坏现有页面拥有数据和请求路径的约定。
