# 5 Minute Demo

这份文档用于销售、客户沟通或管理层演示。

目标不是解释仓库实现细节，而是在 5 分钟内讲清楚：

1. Fast Service Platform 已经能产出什么
2. 这个 demo 不是静态原型，而是可操作的 baseline 企业管理应用
3. 这个 baseline 来自平台复用能力，而不是单独手工开发的样板

## 演示前准备

先按 [GUIDE.md](/home/code/Code/fast-service-platform/demo/GUIDE.md) 启动 demo。

建议确保这两个地址可用：

- frontend: `http://127.0.0.1:<vite-port>`
- backend proxy path: `http://127.0.0.1:<vite-port>/service/*`

## 5 分钟节奏

### 0:00 - 0:30

开场建议：

“这不是一个单独定制项目，而是 Fast Service Platform 基于统一组件和装配路径生成出来的 baseline demo。它代表平台当前可稳定复用的企业管理基线。”

要点：

- 这是平台生成出来的 baseline
- 不是静态页面
- 不是纯 mock

### 0:30 - 1:20

打开 `Dashboard`。

建议说法：

“首页展示的是当前 baseline 的最小企业管理面。这里能看到用户、项目、角色权限、工单和看板这些核心指标，它们都来自真实 backend 服务。”

要点：

- 强调是完整 admin shell
- 强调数据来自真实 backend
- 强调 baseline 已经覆盖企业内部管理的最小工作面

### 1:20 - 2:00

打开 `Users` 和 `Roles`。

建议说法：

“平台的必选核心不是只读展示，而是可管理的身份与权限基线。这里可以直接创建用户、角色、权限，并完成用户和角色的绑定。”

建议动作：

- 展示默认管理员
- 现场新建一个用户
- 如时间允许，新建一个角色或权限

要点：

- 强调 `user-management + RBAC` 是平台必选核心
- 强调不是诊断页，而是可操作工作流

### 2:00 - 3:10

打开 `Projects`。

建议说法：

“在 baseline 里，项目既是业务交付范围，也是后续工程协作的基础锚点。默认 demo data 已经提供了一个项目，我们也可以现场再创建一个。”

建议动作：

- 展示默认 `FSP` 项目
- 现场创建一个新项目

如果要演示 Git：

“如果连接本地仓库，这里还可以展示项目绑定仓库、查看分支上下文和切分支能力。”

要点：

- 强调项目是后续工单、看板的作用域
- 如果没有准备本地 Git 仓库，就不要临场尝试 Git 部分

### 3:10 - 4:10

打开 `Kanban` 和 `Tickets`。

建议说法：

“baseline 不只到项目层，它已经覆盖最小交付工作流。默认 demo data 里已经有一个 board 和两条工单，我们也可以继续创建新的 board 和 ticket，并推进状态。”

建议动作：

- 先展示默认 `Delivery Board`
- 先展示默认 `FSP-1`、`FSP-2`
- 现场创建一个 ticket 或推进一个 ticket 状态

要点：

- 强调最小交付链路已经贯通
- 强调可以从 baseline 继续扩展，而不是从零开始

### 4:10 - 5:00

回到 `Dashboard`。

建议说法：

“刚才的写操作已经回流到首页，这说明这个 baseline 是一个真实可运行、可写入、可扩展的管理应用骨架。平台的价值不只是展示一套页面，而是可以把这套 baseline 作为可复用起点，继续派生新的企业内部管理系统。”

收尾要点：

- 这已经是可演示、可运行、可验证的 baseline
- 这套 baseline 来自平台 assembly path
- 后续新应用不是重新做一遍，而是在这个平台基线上装配和派生

## 演示时不要这样讲

- 不要说“这是一个前端原型”
- 不要说“这里的数据是我临时写死的”
- 不要一开始就讲太多 contract、schema、upgrade 细节
- 不要在没准备本地 Git 仓库时现场尝试 Git repository binding

## 演示时可以这样总结

一句话版：

“Fast Service Platform 已经能稳定提供一个可运行、可操作、可派生的企业管理 baseline，而这个 demo 就是平台当前能力的直接证明。”

客户版：

“我们不是从空白项目开始做每一个内部系统，而是先拿到一个已经具备用户、权限、项目、工单和看板基础能力的 baseline，再按需求继续派生。”
